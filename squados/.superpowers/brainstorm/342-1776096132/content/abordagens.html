Como estruturar o banco de dados?

3 abordagens para o redesign — com trade-offs. Recomendo a B.

A

Reaproveitar tabela atual

Adicionar sector_id na tabela production_processes existente e criar uma nova tabela user_process_assignments para as atribuições.

Prós

  • Menos migration SQL
  • Mantém dados antigos

Contras

  • Mistura catálogo com dados legados
  • Campo assigned_to antigo fica órfão
B ★

Tabelas novas + migrar dados existentes

Criar process_catalog (catálogo global com sector_id) e user_process_assignments (vínculo usuário ↔ catálogo com order e color). Migrar processos antigos para o catálogo automaticamente.

Prós

  • Modelo limpo e sem ambiguidade
  • Fácil de expandir
  • Dados legados migrados

Contras

  • Migration um pouco maior
C

assigned_to nullable (catálogo = NULL)

Usar a mesma tabela: quando assigned_to = NULL é catálogo, quando tem valor é do fluxo do usuário. Adicionar FK catalog_process_id para referência.

Prós

  • Minimal de SQL

Contras

  • Modelo confuso
  • RLS mais complicada
  • Difícil de manter

Esquema recomendado (Abordagem B)

process_catalog
  id, sector_id (FK), title, description, color
  created_by, is_active, created_at, updated_at

process_catalog_media   (igual production_media atual)
  id, catalog_process_id (FK), type, url, caption, order_index

user_process_assignments
  id, user_id (FK), catalog_process_id (FK)
  order_index, color, created_by, created_at
